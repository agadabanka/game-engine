GAME ENGINE

Skills and Machines — what the AI decides vs what the code does

Eleven games now run on one rig. This edition reads the engine through a single lens: which capabilities live as skills (recipes Claude runs by hand) and which have hardened into programmatic machines (code that runs itself the same way every time) — and how to keep moving the line.

Chapter 1 · the state of things

One rig, eleven games, six genres

It started as two games — the-platformer and Jazz — and a promise: don't rebuild the bottom. It is now a constellation: a brawler, a runner, a vertical climber, a shooter, two car-RTSs, a builder, an axolotl swimmer, an astronaut platformer — all off one engine.

What makes a tenth game cheap is that almost everything is inherited, not rebuilt. A new game is mostly a re-skin of a shared machine, plus a few hundred lines of its own level data and identity.

Two materials make up the rig

  • Machines — deterministic code every game runs verbatim: the Studio SDK, the scaffolder, the eval gate, the recorder, the hub.
  • Skills — prose recipes in .claude/skills/ that Claude reads and performs: scaffold a game, make a game end-to-end, cut a trailer, rebuild this book.

The health of the engine is the ratio between them, and which way it's moving.

Chapter 2 · a machine in its purest form

The button

The cleanest machine in the repo is the one that makes the next game. It used to be a checklist a human followed. Now it is one command:

node scripts/new-game.mjs "My Game" --local --hero "…" --verb "…"

  • clone the de-branded base from engine/game-template/
  • rebrand package, README, a fresh DIARY.md
  • write GAME_META.json so the hub shows a rich card
  • create the GitHub repo, push, register with the hub

This is the lesson in miniature: a recipe that stopped changing got frozen into code. Now it can't drift, runs in seconds, and is the same for game one and game eleven. The whole book is about doing this on purpose, everywhere it's safe.

Chapter 3 · the lens

Recipes and machines

Every capability in this system lives in one of two forms, and they behave very differently.

A skill
prose · run by Claude
  • A recipe in SKILL.md the model reads and performs.
  • Flexible — adapts to a new genre, a weird game, a fresh idea.
  • Re-judged every run — so it can drift, vary, or mis-step.
  • Costs model time; quality tracks the model.
A machine
code · runs itself
  • A tool/script/SDK function with a fixed contract.
  • Deterministic — identical inputs, identical output, forever.
  • Testable & fast — an eval can guard it; it runs in seconds.
  • Inherited verbatim by every game; no judgment spent.

Skills are where a capability lives while it's still being figured out. Machines are where it goes once the recipe stops surprising you. The art is knowing which is which — and moving capabilities across the line as they mature.

Chapter 4 · the inventory

Where the line sits today

A faithful map of the engine as it stands — every major capability sorted by its current form.

Machines today
programmatic · in code
  • Studio SDK — engine/sdk/studio.js: Materials, Textures, Toon rigs, Brawl, RNG, Juice, Audio, Cam, the harness.
  • The gate — per-game eval.mjs: determinism + win + non-black, both renderers.
  • The fun model — Studio.Brawl.fun: action/flow/arc/closeness/variety, scored.
  • Headless sim — Studio._headless: ~70× faster seed-scans.
  • Scaffolder — new-game.mjs. Recorder/shorts — record · make-shorts · host-shorts · wire-shorts. Hub — server · aggregate · store.
Skills today
recipes · run by Claude
  • new-game — drive the scaffolder, then finish the deploy.
  • make-game — the whole concept→shipped arc, by hand, stage by stage.
  • trailer — assemble a montage across games.
  • book — author + rebuild this PDF.
  • and the judgment that can't be code: the concept, art direction, what a level should feel like, whether it's fun.

The mechanics have largely crossed into code. What remains a skill is orchestration (the make-game pipeline still runs stage-by-stage by hand) and creative judgment (which should never be code).

Chapter 5 · the direction of travel

Forging skills into machines

The engine matures by one repeated move: when a step in a skill stops needing judgment, harden it into a tool. Things that have already crossed:

was skill  “follow the scaffold checklist”
→
new-game.mjs  machine
was skill  “record each level, mux music, upload”
→
record · make-shorts · host-shorts  machine
was skill  “judge if it's fun”
→
Studio.Brawl.fun (scored)  machine
was skill  “scan seeds for a good match”
→
?r=headless + scan loop  machine
was skill  “deep-link a level per game”
→
?level=N in the template  machine

What should cross next

  • The make-game orchestration → a ship-game.mjs that runs the deterministic stages (scaffold · gate · deploy · record · host · hub-wire) as one pipeline; the skill keeps only the creative top.
  • Art & music passes → engine-level tools the scaffolder installs (a per-game brief in, assets out) instead of re-written per game.
  • Diary · screenshots · registration → a publish.mjs that captures stills, writes the scorecard, and registers the hub card.

Chapter 6 · the machine above the games

Mission control

One tier sits above every game and runs none of them — it observes. Because each game exposes the same surface, the hub pulls them all onto one board.

  • /api/meta — hero, verb, worlds, level count, scorecard, videos, shorts
  • /api/diary — the build log, parsed into entries
  • /api/notes — live playtest feedback, open vs. closed

A storefront, not a spreadsheet

The board is now visual: a gameplay-cover card per game with one coherent button system, a deep-detail modal, and a vertical shorts feed — 23 highlight reels streamed same-origin through the hub's video proxy so private-repo assets play everywhere. Pull-model throughout: a game that isn't deployed still shows up from its GitHub repo.

Chapter 7 · why the split matters

From "I felt something" to "it's fixed"

The reason to push mechanics into machines is to free the skill — Claude's judgment — for the part only it can do: feeling whether a game is good and changing it until it is.

  • Feel it — pause in-game, pin the spot, leave a note.
  • File it — each note becomes a GitHub issue on session start.
  • Fix it — work the issues; the gate guards every change.
  • Record it — the why goes in the diary; the hub shows it.

The owner gives a concept; the machines do the deterministic labour; the skill spends its judgment on art, feel, and fun. Every capability moved into code is one less thing to re-decide — and one more thing that can't break twice.
