Chapter 1

Everything into one

Two games came before this: the-platformer, a working Phaser game on a deploy-ready skeleton, and Jazz, a cartoon run-and-gun built on top of it. Building the second proved the same engine and discipline could carry a brand-new skin and verb.

The lesson Jazz left behind: don't keep rebuilding the bottom. Lift the platform, the engine, and the evaluation tiers into one place, so the third game — and the tenth — starts wired to all of it on day one.

So this repo is two things

  • The new-game button — one command turns "I want to build X" into a fresh, playable, deploy-ready game repo.
  • The mission-control hub — one deployed dashboard that watches every game's live notes, build diaries, and meta.

Not a framework you learn. A rig you inherit.

Chapter 2

The button

One command, six steps, a new game on GitHub and on the board:

node scripts/new-game.mjs "My Game" --hero "a robot ninja" --verb "dash · slash"

  • clone the proven base — a complete, playable platformer
  • rebrand name, description, README; start a fresh build diary
  • write GAME_META.json so the hub shows a rich card
  • create the GitHub repo and push (fresh history)
  • register it with the hub — it appears on mission control at once

What the new game already has

Platform (server · store · Gemini · Lyria) · the Phaser engine (scenes · materials · the level DSL · campaign merge) · the canvas level builder · the full evaluation suite (0-death gate · felt-fun · recorder · vision judge · game-diff) · the notes→diary→issues loop. You don't rebuild any of it — you re-skin the surface (hero, verb, art, music, worlds) and design levels.

Chapter 3

Mission control

The one tier that's new here: a layer above all games. It doesn't run any game — it observes them.

Because every game built on the engine exposes the same surface, the hub pulls them all into one board with a single shape:

  • /api/notes — live playtest feedback (open vs. closed)
  • /api/diary — the build log, parsed into entries
  • /api/meta — hero, verb, worlds, level count, art, music, diff scores

Pull-model, so nothing else has to change

Existing games (the-platformer, Jazz) work unchanged. And a game that isn't deployed yet still shows up — the hub reads DIARY.md and GAME_META.json straight from its GitHub repo. The result: per-game cards, one aggregated notes-triage feed, and the combined diary timeline — the state of every game at a glance.

Chapter 4

From "I felt something" to "it's fixed"

The engine closes the loop that usually breaks: the gap between a vague feeling while playing and an actual change in the code.

  • Feel it — pause in-game, pin the exact spot, leave a note.
  • File it — on session start, each note becomes a GitHub issue.
  • Fix it — work the issues; the 0-death gate guards every change.
  • Record it — the why goes in the build diary; the hub shows it.

One person can run all of it because the robot QA team — the gate, the felt-fun model, the vision judge — does the playtesting that used to take a room of people.

The fast-follow

Today the scaffolder's base is the-platformer. Next, a de-branded copy of the engine lifts into this repo, so game-engine owns the base outright and a new game depends on nothing else.
